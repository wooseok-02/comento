# 프로젝트 기획/설계 워크플로우

이 문서는 어떤 프로젝트를 진행하더라도 코드 작성 전에 기획, 요구사항, 기능, 화면, 데이터, 시스템 흐름을 차례대로 정리하기 위한 범용 작업 흐름이다.

목표는 기능을 빠르게 나열하는 것이 아니라, 사용 목적과 운영 상황을 먼저 이해한 뒤 개발 범위를 명확하게 정하는 것이다.

## 전체 흐름 개요

### 1. 요구사항 정의

- 프로젝트나 기능이 왜 필요한지 정리한다.
- 누가 이 기능을 사용하는지 정리한다.
- 사용자가 어떤 상황에서 어떤 행동을 해야 하는지 정리한다.
- 시스템이 그 행동에 대해 어떤 결과를 만들어야 하는지 정리한다.
- 기능이 운영될 때 생길 수 있는 문제를 미리 생각한다.

### 2. 기능 정의

- 합의된 요구사항을 바탕으로 기능 목록을 작성한다.
- 기능 목록은 세부 구현이 아니라 시스템이 제공해야 할 기능 단위로 묶는다.
- 각 기능별 사용자 흐름을 작성한다.
- 각 기능의 입력값과 결과값을 정리한다.
- 이 내용을 바탕으로 기능 명세서를 작성한다.

### 3. 화면 설계

- 사용자가 기능을 어떤 화면에서 사용할지 정리한다.
- 화면별 주요 UI 요소를 정리한다.
- 사용자가 입력해야 하는 값과 확인해야 하는 값을 구분한다.
- 불필요한 화면 요소가 들어가지 않도록 기능 목적과 연결해서 판단한다.

### 4. 데이터 설계

- 기능을 수행하기 위해 저장해야 하는 데이터를 정리한다.
- 데이터 항목, 필드명, 타입, 필수 여부를 정리한다.
- 데이터가 어디에서 생성되고 어디에서 사용되는지 정리한다.
- 나중 기능과 연결될 수 있는 데이터는 확장 가능성을 고려한다.

### 5. 시스템 설계

- 사용자의 요청이 시스템 내부에서 어떤 순서로 처리되는지 정리한다.
- 주요 처리 단계와 각 단계의 책임을 나눈다.
- 외부 API, 파일 저장소, DB, 벡터 DB 같은 의존성을 정리한다.
- 기능을 어떤 모듈이나 파일로 나눌지 정리한다.

### 6. 예외 상황 정리

- 사용자가 잘못 입력했을 때의 처리를 정리한다.
- 파일, 네트워크, DB, 외부 API 처리 실패 상황을 정리한다.
- 중복 데이터, 권한 문제, 삭제 문제처럼 운영 중 생길 수 있는 상황을 정리한다.
- 실패 시 사용자에게 어떤 메시지를 보여줄지 정리한다.

### 7. 개발 계획 수립

- 어떤 기능부터 개발할지 순서를 정한다.
- 1차 프로토타입에 포함할 범위와 나중으로 미룰 범위를 구분한다.
- 기능 간 의존관계를 확인한다.
- 사용자 승인 후 실제 코드 작성으로 넘어간다.

## 기능 설계 시 Codex가 반드시 짚어야 할 내용

기능을 설계하거나 개발 방향을 제안할 때 Codex는 아래 내용을 명확히 설명한다.

- 이 기능이 왜 필요한지
- 사용자가 어떤 흐름으로 사용하는지
- 어떤 데이터가 저장되는지
- 어떤 파일이나 모듈이 필요할지
- 나중에 다른 기능과 어떻게 연결되는지
- 개발 중 고려해야 할 점
- 하면 안 되는 구현 방식 또는 위험한 선택지
- 나중 단계로 미뤄도 되는 기능은 무엇인지
