# 1. 요구사항 정의 플로우

요구사항 정의는 기능을 바로 떠올리는 단계가 아니다.
먼저 기능이 운영되는 상황을 상상하고, 그 상황에서 생길 수 있는 필요와 문제를 찾아내는 단계다.

## 요구사항 작성 기본 구조

요구사항은 아래 흐름으로 작성한다.

```text
필요성
↓
사용자 행동
↓
시스템 결과
```

### 필요성

- 왜 이 요구사항이 필요한지 설명한다.
- 단순히 기능이 있으면 좋다는 수준이 아니라, 없으면 어떤 문제가 생기는지 생각한다.
- 운영 중 생길 수 있는 위험, 불편, 혼란, 데이터 품질 문제를 함께 고려한다.

### 사용자 행동

- 사용자가 실제로 무엇을 하는지 설명한다.
- 사용자의 역할을 명확히 한다.
- 관리자, 일반 사용자, 승인자, 검토자처럼 역할이 다르면 요구사항도 다르게 본다.

### 시스템 결과

- 사용자의 행동 뒤에 시스템이 무엇을 저장, 변경, 표시, 차단, 처리해야 하는지 설명한다.
- 결과는 사용자에게 보이는 결과와 내부적으로 처리되는 결과를 구분해서 생각한다.
- 데이터가 나중에 다른 기능에서 쓰일 가능성도 함께 고려한다.

## 요구사항을 생각하는 질문

요구사항을 작성할 때는 아래 질문을 순서대로 던진다.

```text
1. 이 기능은 누가 쓰는가?
2. 그 사람은 왜 이 기능을 쓰는가?
3. 사용자가 실수하거나 곤란해지는 지점은 어디인가?
4. 시스템이 그 문제를 막거나 줄이려면 무엇을 해야 하는가?
5. 이 기능이 만드는 데이터는 나중에 어디에 쓰이는가?
6. 지금 만들 범위와 나중으로 미룰 범위는 무엇인가?
```

## 요구사항 작성 팁

- 단순히 기능을 상상하지 않는다.
- 기능이 운영될 때 생길 문제를 미리 상상한다.
- 사용자가 말하지 않은 입력값, 화면 요소, 데이터 항목을 임의로 확정하지 않는다.
- 구현 방식은 요구사항 단계에서 확정하지 않는다.
- 다만 나중 설계에 영향을 줄 수 있는 가능성은 요구사항에 남긴다.
- “반드시 구현한다”와 “나중에 활용될 수 있다”를 구분해서 쓴다.

## 요구사항에서 하면 안 되는 것

- 기능 이름만 보고 바로 화면이나 DB 구조를 정하지 않는다.
- 사용자가 입력해야 할 항목을 임의로 정하지 않는다.
- 내부 구현 방식을 요구사항처럼 확정하지 않는다.
- 나중 기능까지 한 번에 섞어서 1차 범위를 키우지 않는다.
- 편의를 이유로 위험한 운영 방식을 당연하게 받아들이지 않는다.

## 요구사항 예시 형식

```text
요구사항
관리자는 파일 업로드 시 문서 카테고리를 지정할 수 있다.

필요성
공통 자료가 많아지면 문서의 성격에 따라 구분해서 관리해야 하며, 향후 검색 필터링과 검색 품질 개선에도 활용될 수 있다.

사용자 행동
관리자는 문서를 업로드할 때 정해진 카테고리 목록 중 하나를 선택한다.

시스템 결과
시스템은 업로드된 문서와 선택된 카테고리를 함께 저장한다.
```

## 요구사항 단계에서 확정할 것과 미룰 것

### 지금 확정해도 되는 것

- 누가 사용하는 기능인지
- 어떤 상황에서 사용하는 기능인지
- 사용자가 어떤 행동을 해야 하는지
- 시스템이 어떤 결과를 만들어야 하는지
- 1차 프로토타입에 포함할 범위
- 나중으로 미룰 범위

### 나중 설계 단계로 미룰 것

- DB 테이블 구조
- 함수와 클래스 구조
- 파일/모듈 구조
- 라이브러리 세부 사용 방식
- 벡터 검색 알고리즘의 세부 전략
- UI 배치와 컴포넌트 구성
