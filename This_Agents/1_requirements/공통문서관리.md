## 이번 프로젝트에서 정리한 요구사항 예시

### 요구사항 1. 공통 자료 관리는 관리자만 사용한다

필요성:
모든 사용자가 공통 자료를 직접 업로드하면 잘못된 정보가 AI 답변의 근거로 사용될 위험이 있다.

사용자 행동:
관리자는 검증된 내부 문서를 공통 자료 관리 기능에서 등록, 조회, 삭제한다.

시스템 결과:
등록된 문서는 모든 사용자의 AI 질의응답과 RAG 검색에 활용되는 공통 벡터 DB의 기반 자료가 된다.

### 요구사항 2. 1차 공통 자료는 파일 기반 문서만 대상으로 한다

필요성:
공통 자료는 관리자가 검증한 고정 문서를 기반으로 해야 한다.
URL은 내용 변경, 접근 실패, 본문 추출, 갱신 주기 같은 별도 고려사항이 많다.

사용자 행동:
관리자는 PDF, TXT, MD 파일을 업로드한다.

시스템 결과:
시스템은 업로드된 파일만 공통 자료 처리 대상으로 삼고, URL 수집은 이후 웹 리트리버 기능에서 별도로 다룬다.

### 요구사항 3. 관리자는 파일 업로드 시 카테고리를 지정한다

필요성:
공통 자료가 많아지면 문서의 성격에 따라 구분해서 관리해야 한다.
카테고리는 향후 RAG 검색에서 출처 표시, 검색 필터링, 검색 품질 개선을 위한 메타정보로 활용될 수 있다.

사용자 행동:
관리자는 문서를 업로드할 때 정해진 카테고리 중 하나를 선택한다.

시스템 결과:
시스템은 업로드된 문서와 선택된 카테고리를 함께 저장한다.

### 요구사항 4. 문서 제목은 파일명에서 자동 생성한다

필요성:
1차 프로토타입에서는 관리자가 입력해야 할 항목을 줄여 문서 등록 과정을 단순하게 만든다.

사용자 행동:
관리자는 파일을 선택하고 카테고리를 지정한 뒤 업로드한다.

시스템 결과:
시스템은 업로드한 파일명에서 확장자를 제외한 값을 문서 제목으로 사용한다.

### 요구사항 5. 파일 업로드 시 자동으로 벡터 DB 저장 절차를 수행한다

필요성:
공통 자료 관리의 핵심 목적은 단순 문서 보관이 아니라, 생성형 AI가 활용할 수 있는 벡터 DB를 구축하는 것이다.

사용자 행동:
관리자는 파일을 업로드하고 처리 완료 메시지를 확인한다.

시스템 결과:
시스템은 문서 업로드 후 자동으로 텍스트 추출, 청킹, 임베딩, 벡터 DB 저장을 수행한다.

### 요구사항 6. 중복 문서는 파일 내용 해시 기준으로 확인한다

필요성:
같은 내용의 문서가 여러 번 벡터 DB에 저장되면 검색 결과와 AI 답변 근거가 중복되어 품질이 떨어질 수 있다.

사용자 행동:
관리자가 파일을 업로드한다.
이미 같은 내용의 문서가 등록되어 있으면 시스템은 업로드를 중단하고 중복 안내 메시지를 보여준다.

시스템 결과:
시스템은 업로드된 파일의 내용 해시를 생성하고 기존 문서의 해시와 비교한다.
같은 해시가 있으면 원본 파일 저장, 텍스트 추출, 임베딩, 벡터 DB 저장을 진행하지 않는다.

### 요구사항 7. 관리자는 문서 처리 상태를 확인할 수 있다

필요성:
파일 업로드가 성공해도 텍스트 추출, 청킹, 임베딩, 벡터 DB 저장 중 실패할 수 있다.

사용자 행동:
관리자는 파일 업로드 후 처리 완료, 실패, 중복 상태를 확인한다.

시스템 결과:
시스템은 문서별 처리 상태를 저장하고 화면에 표시한다.

### 요구사항 8. 문서 삭제는 비활성 처리로 수행한다

필요성:
잘못 업로드된 문서나 더 이상 사용하지 않는 문서가 AI 답변의 근거로 사용되지 않도록 관리해야 한다.
다만 완전 삭제하면 나중에 이력 추적이 어려워질 수 있다.

사용자 행동:
관리자는 문서 목록에서 삭제할 문서를 선택하고 삭제를 요청한다.

시스템 결과:
활성 상태
- AI 검색과 벡터 DB 재생성 대상에 포함

비활성 상태
- AI 검색과 벡터 DB 재생성 대상에서 제외
- 원본 파일과 문서 기록은 유지

### 요구사항 9. 관리자는 현재 활성 문서를 기준으로 벡터 DB를 재생성할 수 있다.

필요성
문서의 활성/비활성 상태가 변경되거나 벡터 DB 상태가 꼬였을 때, 현재 사용 가능한 문서 기준으로 AI 검색 데이터를 다시 맞출 수 있어야 한다.

사용자 행동
관리자는 문서 관리 화면에서 벡터 DB 재생성 버튼을 클릭한다.

시스템 결과
시스템은 metadata에서 활성 상태이고 처리 완료된 문서만 대상으로 다시 텍스트 추출, 청킹, 임베딩을 수행하고 벡터 DB를 재생성한다.
