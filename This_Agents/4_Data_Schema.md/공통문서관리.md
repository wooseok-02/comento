# 설명 : 각 기능이 동작하려면 어떤 데이터가 필요하고, 그 데이터가 어디에서 만들어져 어디에 쓰이는지 정하는 단계

# 공통 자료 관리 데이터 설계

공통 자료 관리는 1차 프로토타입에서 아래 3가지 데이터를 관리한다.

```text
1. 문서 자체 저장
2. 문서에 대한 metadata 저장
3. 문서의 벡터 내용 저장
```

구체적인 파일명, 내부 저장 형식, LangChain/FAISS 저장 구조는 구현 단계에서 사용하는 라이브러리 방식에 맞춘다.

---

## 1. 문서 자체 저장

관리자가 업로드한 원본 문서를 저장한다.

### 저장해야 하는 이유

- 관리자가 원본 문서를 다시 다운로드할 수 있어야 한다.
- 문서를 다시 벡터화하거나 재처리할 때 원본이 필요하다.
- AI 답변의 근거가 되는 원본 자료로 연결할 수 있어야 한다.

### 필요한 정보

```text
원본 파일
- 관리자가 업로드한 실제 PDF, TXT, MD 파일

저장 위치
- 원본 파일을 다시 찾고 다운로드하기 위해 필요

파일 형식
- PDF, TXT, MD 중 어떤 문서인지 판단하기 위해 필요
- 텍스트 추출 방식을 결정하는 데 필요
```

---

## 2. 문서 metadata 저장

문서를 관리하고 조회하기 위한 정보를 저장한다.

### 저장해야 하는 이유

- 문서 목록을 조회할 수 있어야 한다.
- 카테고리별로 문서를 구분할 수 있어야 한다.
- 중복 문서 업로드를 막을 수 있어야 한다.
- 문서의 처리 상태를 확인할 수 있어야 한다.
- 활성/비활성 상태를 관리할 수 있어야 한다.
- 원본 파일 다운로드와 벡터 데이터 연결이 가능해야 한다.

### 필요한 필드

```text
document_id
- 문서를 고유하게 식별하기 위해 필요
- 원본 문서와 벡터 데이터를 연결하기 위해 필요

file_name
- 화면에 문서 이름을 표시하기 위해 필요
- 다운로드 시 사용자에게 보여줄 이름으로 필요

category
- 문서를 규정, 내규, 법령, 판례, 보도자료 등으로 구분하기 위해 필요
- 문서 조회와 향후 검색 필터링에 활용하기 위해 필요

file_type
- 파일이 PDF, TXT, MD 중 무엇인지 구분하기 위해 필요
- 텍스트 추출 방식을 결정하기 위해 필요

file_path
- 원본 파일의 저장 위치를 찾기 위해 필요
- 다운로드와 재처리를 위해 필요

file_hash
- 파일 내용 기준으로 중복 문서를 판단하기 위해 필요

status
- 문서가 처리중, 완료, 실패, 중복 중 어떤 상태인지 표시하기 위해 필요

status_message
- 현재 처리 단계나 실패 사유를 사용자에게 보여주기 위해 필요

is_active
- 해당 문서를 AI 검색에 사용할지 제외할지 판단하기 위해 필요

created_at
- 문서를 최신순으로 조회하기 위해 필요
- 업로드 시점을 확인하기 위해 필요
```

## 3. 문서의 벡터 내용 저장

문서를 RAG 검색에 사용할 수 있도록 벡터화한 내용을 저장한다.

### 저장해야 하는 이유

- 사용자 질문과 관련 있는 문서 내용을 검색하기 위해 필요하다.
- AI 답변 생성 시 참고할 근거 문서를 찾기 위해 필요하다.
- 답변에 출처를 연결하기 위해 필요하다.

### 필요한 정보

```text
문서 청크
- 긴 문서를 검색 가능한 단위로 나누기 위해 필요

청크 임베딩
- 사용자 질문과 의미적으로 가까운 내용을 찾기 위해 필요

document_id
- 검색된 벡터가 어떤 원본 문서에서 나온 것인지 연결하기 위해 필요

chunk_id
- 검색된 청크를 구분하기 위해 필요

chunk_index
- 원본 문서 안에서 청크의 순서를 파악하기 위해 필요

file_name
- AI 답변 출처에 문서명을 표시하기 위해 필요

category
- 향후 카테고리 기반 검색이나 필터링에 활용하기 위해 필요

source_path
- 검색 결과에서 원본 문서로 연결하기 위해 필요

is_active
- 비활성 문서를 검색 결과에서 제외하기 위해 필요
```

---

## 설계 원칙

- 문서 자체, 문서 metadata, 문서 벡터 내용은 역할이 다르므로 분리해서 생각한다.
- 구체적인 저장 파일명이나 내부 구조는 구현 단계에서 사용하는 라이브러리 방식에 맞춘다.
- FAISS 또는 LangChain 내부 저장 형식은 설계 단계에서 임의로 고정하지 않는다.
- 프로토타입에서는 정식 DB를 쓰지 않더라도, 어떤 데이터가 왜 필요한지는 명확히 한다.
- 문서 metadata는 문서 조회, 중복 확인, 상태 표시, 활성/비활성 관리의 기준이 된다.
- 벡터 내용은 AI 검색과 답변 출처 연결을 위한 데이터로 본다.
